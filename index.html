Solo Developer · C++ / DirectX 12
独立开发者 · C++ / DirectX 12

C++DirectX 123D RenderingAI FSMCollision DetectionHUD

A first-person 3D shooter built from scratch in C++ and DirectX 12. Covers the full rendering pipeline from swap chain setup through scene management, player input, enemy AI states, collision detection, and basic gameplay loop.

使用 C++ 与 DirectX 12 从零构建的第一人称 3D 射击游戏，完整覆盖从交换链设置到场景管理、玩家输入、敌人 AI 状态、碰撞检测与基础玩法循环。

Key Systems 核心系统
  • Mouse look, WASD movement, shooting and reload.
  • Dinosaur enemy FSM: patrol → chase → melee attack.
  • Ammo, health, and AABB collision systems.
  • Screen-space HUD independent of world render pass.
  • 鼠标视角、WASD 移动、射击与装弹。
  • 恐龙敌人 FSM：巡逻→追踪→近战攻击。
  • 弹药、生命值与 AABB 碰撞系统。
  • 独立于世界渲染 pass 的 screen-space HUD。
Reflections 技术反思
  • Enemy pathfinding is grid-based; A* would improve navigation in open spaces.
  • Collision response is impulse-only; adding a physics integrator would help with slopes.
  • Texture streaming not yet implemented — all assets load upfront.
  • 敌人寻路基于网格，开放空间中 A* 效果更优。
  • 碰撞响应仅用冲量，添加物理积分器能更好处理斜面。
  • 暂未实现贴图流式加载，所有资产目前预加载。

2D Bullet-Hell Survival Game

2D 弹幕生存游戏 Demo

Solo Developer · C++ Gameplay Systems
独立开发者 · C++ 玩法系统
2025.10 — 2025.11
C++2D GameSave SystemItem SystemWave Design

A C++ bullet-hell survival game with WASD movement, area-attack skills, random item rewards, complete save/load, and escalating wave difficulty across two map modes.

基于 C++ 的弹幕生存游戏，实现 WASD 移动、范围攻击、随机道具奖励、完整存档读档，以及两种地图模式下的递增波次难度。

Key Systems 核心系统
  • WASD movement and radial area-attack with cooldown.
  • Binary checkpoint save/load with slot selection.
  • Weighted random item pool with stat modifiers.
  • Enemy wave manager with configurable spawn curves.
  • WASD 移动与带冷却的环形范围攻击。
  • 带槽位选择的二进制存档/读档。
  • 带权重的随机道具池与属性加成。
  • 可配置生成曲线的波次管理器。
Reflections 技术反思
  • Wave balance tuned by hand; a data-driven config file would make iteration faster.
  • Item pool is fixed — a weighted rarity tier system would scale better.
  • No spatial partitioning; bullet vs enemy checks are O(n²) above 200 entities.
  • 波次平衡靠手动调参，数据驱动配置文件会让迭代更快。
  • 道具池固定，稀有度权重分层系统扩展性更好。
  • 无空间分区，200 个实体以上时子弹碰撞检测为 O(n²)。

Frontend Engineer · Team of 4
前端工程师 · 四人团队
2023.09 — 2023.12
HarmonyOSArkUISpringBootFrontend PerformanceTeam

A Xiaohongshu-style campus social app built with HarmonyOS ArkUI and SpringBoot. I owned the frontend architecture — component structure, interaction standards, and rendering-performance work to smooth list scrolling.

使用 HarmonyOS ArkUI + SpringBoot 开发的类小红书校园社交应用。我负责前端架构设计——组件结构、交互规范，以及针对列表滚动的渲染性能优化。

Key Contributions 主要贡献
  • Designed the component hierarchy and inter-page navigation model.
  • Established team-wide interaction and spacing standards.
  • Reduced re-renders via keyed reconciliation; improved scroll smoothness.
  • 设计了组件层级结构与页面间导航模型。
  • 制定了团队统一的交互与间距规范。
  • 通过 keyed 协调策略减少重渲染，提升列表滑动流畅度。
Reflections 技术反思
  • Lacked a shared state layer — prop drilling between nested components added friction.
  • API contracts defined late in the project, causing frontend rework.
  • No automated UI tests; regressions caught only through manual review.
  • 缺乏共享状态层，嵌套组件间的属性传递增加了额外摩擦。
  • API 契约定义较晚，导致前端返工。
  • 无自动化 UI 测试，回归问题只能靠人工审查发现。
